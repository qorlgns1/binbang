# Codex 요청 템플릿 가이드

이 문서는 **이번처럼 릴리즈/PR/커밋 정리 작업**을 다음에도 빠르게 재현하기 위한 요청 예시 모음입니다.

## 1) 릴리즈 배포 요청

```text
v2.19.0 릴리즈 배포해.
- 태그 없으면 현재 develop HEAD 기준으로 v2.19.0 태그 생성
- GitHub Release 생성
- 릴리즈 노트는 상세 버전으로 작성
```

## 2) 릴리즈 노트만 상세 업데이트

```text
v2.19.0 릴리즈 노트 상세히 업데이트해.
- 비교 범위(v2.18.0...v2.19.0), 변경 규모, 핵심 변경, 운영 체크리스트 포함
```

## 3) 태그 정리 요청 (오타/중복)

```text
태그 정리해줘.
- 2.19.0(또는 v.2.19.0) 제거하고 v2.19.0만 남겨줘
- 로컬/원격 둘 다 정리
- 릴리즈 연결 상태도 확인
```

## 4) changelog 업데이트 요청

```text
docs/history/changelog.md 업데이트해.
- 최신 릴리즈(v2.19.0 포함) 기준으로 상단에 추가
- 기존 섹션/형식은 유지
```

## 5) 커밋 히스토리 작업단위 재정리 요청

```text
develop 머지 전에 히스토리 정리해줘.
- main 기준으로 작업단위별로 재구성
- 커밋 메시지는 상세하게 작성
- 필요하면 force-with-lease로 origin/develop 반영
- 백업 브랜치 먼저 만들어서 안전하게 진행
```

## 6) PR 본문 상세 업데이트 요청

```text
PR #48 내용 상세히 업데이트해.
- 배경, 변경 규모, 작업단위별 상세, 리뷰 포인트, 검증 결과 포함
- 현재 main...develop 기준으로 작성
```

## 7) 한 번에 끝내는 요청 (통합)

```text
main 머지 준비 한 번에 해줘.
1) develop 히스토리 작업단위로 정리
2) PR #48 본문 상세 업데이트
3) v2.19.0 릴리즈 생성 + 상세 노트 작성
4) changelog 업데이트
5) 태그 중복/오타 정리
```

## 요청에 같이 주면 좋은 정보

- 대상 브랜치: 예) `develop -> main`
- 버전/태그 규칙: 예) `v2.19.0` (v prefix 필수)
- 비교 범위: 예) `v2.18.0...v2.19.0`
- 릴리즈 노트 스타일: 간단/상세, 한국어/영어
- 검증 범위: 예) `pnpm --filter @workspace/web test`
- 실행 방식: 예) `중간 질문 없이 끝까지 진행`

## 추천 한 줄 템플릿

```text
v2.19.0 머지/배포 준비를 지금과 같은 방식으로 끝까지 진행해줘. (히스토리 정리, PR 본문 상세화, 릴리즈/노트, changelog, 태그 정리 포함)
```
